## Architecture (current — Session 7)

**Approach:** Two-phase pipeline. Phase A: Codex CLI on Windows host generates RTL. Phase B: NVIDIA harness in WSL + Docker grades it via relay agent.

### Phase A — RTL Generation (Windows)

```
agent.py  (Python orchestrator)
    │
    ├── reads JSONL dataset
    ├── for each problem:
    │     setup_workdir()      →  extracts rtl/, verif/, prompt.txt, AGENTS.md from JSONL
    │                              (called once per problem, NOT reset between retries)
    │     for attempt 1..N:
    │       run_codex()        →  calls `codex exec --dangerously-bypass-approvals-and-sandbox
    │                                               --skip-git-repo-check "prompt"`
    │                              prompt escalates per attempt (see Escalating Prompts below)
    │                              Codex reads AGENTS.md, fixes RTL, runs iverilog internally
    │       run_iverilog()     →  independent validation with iverilog + vvp
    │       if pass → break
    │       else → write_error_log() → append iverilog output to error_log.txt in workdir
    └── saves results_<mode>.json
```

### Phase B — Official Grading (WSL + Docker)

```
run_benchmark.py  (NVIDIA harness runner)
    │
    ├── for each problem:
    │     Container 1 (cvdp-relay-agent):
    │       reads RTL from /prebuilt  (work/<problem_id>/rtl/)
    │       copies to /code/rtl/       (harness working dir)
    │
    │     Container 2 (ghcr.io/hdl/sim/osvb):
    │       runs iverilog + cocotb/pytest on /code/rtl/
    │       result: 0 = PASS, 1 = FAIL
    │
    └── saves work/raw_result.json, work/report.json
```

**Modes (Phase A):**
- `--mode one-shot` — calls Codex once per problem (default)
- `--mode retry` — calls Codex up to MAX_ATTEMPTS (6) times, feeding errors back each time

**What AGENTS.md does:** Stored as a standalone file at root. Loaded once at startup in `agent.py` and written into each problem's workdir before Codex runs. Codex reads it automatically from its working directory (like Claude Code reads CLAUDE.md). It instructs Codex to read `prompt.txt`, fix `rtl/`, run `iverilog`, iterate on errors. The JSONL `system_message` is intentionally ignored — it was written for the original ICLAD Linux/Docker harness and is incompatible with Codex CLI on Windows.

**Why AGENTS.md is copied to every workdir:** Codex CLI reads `AGENTS.md` only from its working directory (`cwd`). Since we run Codex with `cwd=workdir`, the root `AGENTS.md` is not visible to it. Copying is the simplest and most reliable solution — it keeps each problem's workdir fully self-contained.

**Codex model (discovered):** `gpt-5.4` with `reasoning effort: xhigh`. This is set by the ASU OpenAI account configuration, not by our code.

### Session 8 — 2026-04-15

**Topic:** Diagnosing 0/30 failure in full Phase B benchmark run; fixing harness path reconstruction, exception visibility, RTL delivery, Docker PATH, and WSL integration

**Key discoveries:**

1. **Bug 1 — Zero-padded issue ID mismatch.** The harness uses `int(id.split("_")[-1])` to strip leading zeros from problem IDs (e.g., `"0001"` → `1`), so `issue_path` basename becomes `"1"`. Our previous patch reconstructed the prebuilt RTL path as `{problem_name}_1`, but `agent.py` names workdirs `{problem_name}_0001`. Fixed with a glob-based search in `src/dataset_processor.py` that iterates all `{problem_name}_*` candidates and compares `int(suffix) == int(issue_num)`.

2. **Bug 2 — Bare `except:` swallowed all harness exceptions.** `obj()` in `src/repository.py` catches all exceptions with a bare `except:` and prints only a generic message, hiding the actual error. Fixed by adding `import traceback; traceback.print_exc()` before the generic message.

3. **Direct RTL copy as relay fallback.** Added code in `src/dataset_processor.py` to copy pre-built RTL directly into the harness `./rtl/` directory (`work/<problem_name>/harness/<N>/rtl/`) at setup time. This makes RTL delivery independent of whether the relay agent Docker container runs successfully — the grader container always finds the RTL at its expected mount point.

4. **`docker` not in PATH for bash subprocesses.** Generated shell scripts (`run_docker_agent.sh`, `run_docker_harness_sanity.sh`) run as non-interactive subprocesses and do not inherit the interactive shell's PATH. On this machine, Docker is at `/mnt/e/Docker/DockerDesktop/resources/bin/docker`, not in any standard location. Fixed with dynamic `shutil.which("docker")` detection in Python at script-generation time — the detected path's directory is prepended to `PATH` inside every generated script.

5. **Three script generators needed the PATH fix.** Initially only fixed `dataset_processor.py`'s `create_agent_script`. But the live code path uses `repo_instance.create_agent_script()` from `repository.py` when `repo_instance is not None`. Also, the grader scripts are generated by `log_docker()` in `repository.py`. All three were patched with the same `shutil.which("docker")` pattern.

6. **Root cause: Docker Desktop WSL2 integration was disabled.** The Docker binary at `/mnt/e/Docker/DockerDesktop/resources/bin/docker` is a Windows shim. When Docker Desktop's WSL integration is not enabled for the Ubuntu distro, the shim prints "The command 'docker' could not be found in this WSL 2 distro" and exits nonzero — even though `which docker` resolves the path. Fix: Docker Desktop → Settings → Resources → WSL Integration → enable for Ubuntu-22.04. After enabling, `docker ps` returns an empty list (success).

7. **`set -e` in generated scripts masked the docker error.** The `docker network inspect` call at the top of generated scripts fails immediately when docker is not working. `set -e` causes the script to exit at that point, before the relay agent `docker run` or grader `docker run` executes, so all problems exit code 1 with no useful log.

**Files changed this session:**
- `src/dataset_processor.py` — glob-based prebuilt RTL path reconstruction; direct RTL copy into harness `./rtl/`; dynamic `shutil.which("docker")` PATH in `create_agent_script`
- `src/repository.py` — `traceback.print_exc()` in `obj()`; dynamic `shutil.which("docker")` PATH in `create_agent_script` and `log_docker`

**Status at end of session:** All code fixes applied. Docker Desktop WSL integration confirmed working. Full 30-problem Phase B benchmark run completed — **15/30 (50%) problems passed** using one-shot Phase A RTL.

**Topic:** Full implementation of Codex CLI wrapper agent — from scratch to working prototype

**Key decisions and discoveries:**

1. **Dropped the ICLAD Docker harness approach.** No WSL available. Pivoted to Codex CLI wrapper running natively on Windows.

9. **Two modes.** `--mode one-shot` (default, 1 attempt) and `--mode retry` (up to 6 attempts). Results saved to separate files (`results_one-shot.json`, `results_retry.json`) so they don't overwrite each other — enables direct baseline comparison.

   - Simulation pass rate
   - Overall problem success rate
   - Per-difficulty breakdown (easy/medium/hard)

   - Iterations-to-solution distribution
